## Known Limitations

These are genuine real-browser/account runtime checks that cannot be proven in jsdom/Node (consistent with as-built.md "Runtime checks deferred"):

1. **QA-53a — Firefox 128 grant-then-copy floor (⊘ Deferred).** The pinned `strict_min_version` "128.0" and the end-to-end combined `chrome.permissions.request` → `navigator.clipboard.writeText` within user activation on the actual pinned Firefox build cannot be exercised in jsdom. Requires a real Firefox 128 run.
2. **Live clipboard activation across the grant-then-copy flow (⊘ Deferred for the cross-browser/activation aspect).** The Chrome path was verified live by the user (visible "Copied to clipboard" banner), but the activation-window timing after the awaited lookup, and the same on Firefox, is a real-browser check. `copyText`'s failure semantics are unit-covered; the live activation success on each engine is not.
3. **OpenWeather One Call 3.0 key activation (⊘ Deferred).** Whether a freshly created OpenWeather key is actually entitled to the One Call 3.0 `timemachine` endpoint is an account/runtime check against the live API; the suite mocks `extFetch` and cannot validate real key entitlement.

Additionally, several criteria are ◐ Partial. QA-42 is Partial for a substantive reason — the implementing service contract is correct but the shipped popup's key gate diverges from the acceptance criterion (a missing OpenWeather key blocks tide too; see Convention Flags). ## Convention Flags

- **OpenWeather key gating in the popup (QA-42 / FR-42) — substantive divergence, not just a flag.** FR-42 requires that a missing OpenWeather key block weather only, while tide MAY proceed. The pure services honor this (`getTide` needs only eBird; NOAA is keyless), but `Popup.tsx:187-194` gates the *entire* lookup on BOTH keys: it pushes `'ebird'` and/or `'openweather'` to `missing`, and if anything is missing it sets `pageState='missing-keys'` and returns before any lookup runs. So in the shipped UI a user with eBird-but-no-OpenWeather sees the missing-keys nudge instead of a tide-only result. This is why QA-42 is recorded as ◐ Partial rather than Pass: the service-level contract (the FR's literal subject) is satisfied, but the rendered v1 behavior fails the acceptance condition. Fix would be to scope the OpenWeather requirement to the weather slot and let the tide lookup proceed on eBird alone.
- **Negative-duration clamp (QA-09 / FR-09):** `weatherService.ts:173` and `tideService.ts:73` use `(duration_hrs || 1)`, which coerces only falsy values (0/NaN/undefined) to 1. A *negative non-zero* `durationHrs` (e.g. -2) is NOT clamped to 1 — it would produce `endTs < startTs`. The PRD says negative duration is "out of contract" and should be "clamped to the default (1)". The code does not explicitly clamp negatives; it relies on negatives being out of contract. Functionally fine for real eBird data (durationHrs is never negative) but not a literal implementation of the clamp wording. No test covers a negative duration.
- **QA-20/QA-51 byte-for-byte vs SnowRaven source:** verification rests on golden oracles and structural fidelity, not a literal `diff` against the upstream SnowRaven repository (not present in `/home/parallels/snowraven-mini`). The goldens encode the Python-oracle output, which is strong evidence, but a true source-identity check was not possible in this workspace.
- **`error` rendered as role=status, not alert (QA-37):** correct per FR-37/FR-37a (the `error`/`unavailable` tide states are intentionally quiet status lines, not alerts). Noted only to confirm it is intentional, not a missed alert role.
