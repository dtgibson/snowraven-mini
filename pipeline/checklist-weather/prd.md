# PRD — Weather and Tide on the eBird Checklist Page
**Feature:** checklist-weather
**Date:** 2026-06-08
**Stage:** 2 — The Planner
**Source:** strategic-brief.md (approved)

## Feature Overview

SnowRaven Mini is a popup-only Manifest V3 browser extension for Chrome and Firefox that brings SnowRaven's weather and tide comment blocks onto the eBird checklist edit page. When a birder editing a checklist clicks the toolbar button, the extension reads the checklist ID from the active tab's URL, resolves that checklist's coordinates and observation date/time through the eBird API (using SnowRaven's exact region-centre-first fallback, not the page's map pin), fetches historical weather from OpenWeather One Call 3.0 `timemachine` and — for US coastal checklists within NOAA's nearest-station range — tide from the keyless NOAA CO-OPS API. It formats both byte-for-byte identical to the SnowRaven desktop app (including the HTML `<a>` attribution), auto-copies only the weather block to the clipboard within the click's user-activation window, and shows weather and tide together in a small popup, each with its own copy button.

The standalone weather and tide blocks that v1 renders and copies join their lines with single `\n` and contain NO blank lines (the only `\n\n`-separated form is the dead-but-faithful `buildCombined` output kept per FR-51, which is unwired in v1). Byte-for-byte parity therefore covers exact line content, order, single-`\n` joins, no trailing newline, and the literal HTML `<a>` attribution — not blank lines.

The entire feature runs on the user's machine: there is no backend, proxy, or hosted service. The user supplies their own eBird and OpenWeather API keys, stored locally; NOAA needs no key. The only network calls are the direct requests to `api.ebird.org`, `api.openweathermap.org`, and `api.tidesandcurrents.noaa.gov`, whose host access is granted on demand from a user gesture. All weather/tide/lookup logic is ported byte-for-byte from SnowRaven's pure TypeScript modules so that output is identical. The one porting gap is timezone resolution: the desktop app derives the IANA timezone from a native library that the extension cannot reuse, so the extension must re-provide an equivalent IANA timezone name in-browser. The product reuses SnowRaven's accessibility standard: WCAG 2.1 AA, keyboard- and screen-reader-operable, OS-driven dark theme, and reduced-motion respected.

## User Stories

- **US-01** — As a birder editing an eBird checklist in my browser, I want one click on the toolbar button to put the SnowRaven weather block on my clipboard, so that I can paste it into my checklist comment without leaving the page.
- **US-02** — As a birder who wants both conditions, I want to see the weather and the tide for my checklist together in the popup, so that I can review them before deciding what to put in my comment.
- **US-03** — As a birder who sometimes wants tide in my comment, I want the tide block to have its own copy button and never be auto-copied, so that I stay in control of whether tide goes into the checklist.
- **US-04** — As a first-time user, I want to paste my two free API keys (eBird and OpenWeather) once on an options page, so that thereafter a single click does the lookup.
- **US-05** — As a birder who uses Chrome on one machine and Firefox on another, I want the extension to behave identically in both browsers, so that I get the same one-click result everywhere I bird.
- **US-06** — As a birder who relies on a keyboard and a screen reader, I want every control and result in the popup to be fully operable and announced, so that I can copy weather and tide without a mouse.
- **US-07** — As a birder, I want clear, recoverable feedback when something is wrong — a non-checklist page, a missing key, an invalid/expired key, a rate limit, a checklist that won't resolve, or a service that fails — so that I always know why nothing copied and what to do next.

## Functional Requirements

### Checklist Page Detection and ID Extraction

- **FR-01** — On the toolbar-click gesture, the popup shall read the active tab's URL via the `activeTab` permission only; it shall not poll, watch, or read tabs at any other time, and shall not use a content script or background worker to observe navigation.
- **FR-02** — The popup shall treat the active tab as an eBird checklist edit page only when its URL has the `https` scheme, the `ebird.org` host, and carries a checklist submission ID that validates per FR-03. Query-parameter order and additional unrelated query parameters shall not defeat detection.
- **FR-03** — The popup shall extract the checklist ID using SnowRaven's `extractChecklistId` rules (trim surrounding whitespace, strip trailing slashes, drop the query string by splitting on `?`, take the last path segment when the remainder contains `/`) and shall accept it only when it matches `/^S\d+$/` exactly (an uppercase `S` followed by one or more digits, no other characters) per `isValidChecklistId`. The extracted, validated ID shall be the single key for all downstream eBird requests and shall be echoed unchanged in the result. (Source: `/home/parallels/snowraven/frontend/src/lib/checklistId.ts:3-11`; backend guard `/home/parallels/snowraven/backend/routers/weather.py:17`.)
- **FR-04** — When the active tab is not a recognizable eBird checklist edit page, or the extracted ID fails validation, or the active-tab URL is unreadable (e.g. `chrome://`, `about:`, other restricted pages), the popup shall show a calm, non-error informational state telling the user to open an eBird checklist edit page, and shall NOT fetch, auto-copy, or request host permissions. This state is distinct from the resolvable-but-no-coordinates case (FR-06a), which is reached only after a valid page and ID are confirmed and the eBird lookup runs.

### Coordinate, Time, and Timezone Resolution (eBird API)

- **FR-05** — On a valid checklist page, the popup shall GET `https://api.ebird.org/v2/product/checklist/view/{checklistId}` with header `X-eBirdApiToken` set to the user's eBird key, and read `obsDt` (observed local time, either `YYYY-MM-DD HH:MM` or date-only `YYYY-MM-DD`), `locId`, `locName` (may be absent), and `durationHrs` (defaulting to `1` when absent or falsy). An `obsDt` longer than 16 characters (e.g. carrying seconds) shall be silently truncated to its first 16 characters (`s.slice(0,16)`) per `normalizeObsDt`; a date-only `obsDt` (length 10) shall be normalized to `<date> 00:00`. An empty or otherwise unparseable `obsDt` shall fall through to the timestamp-parser's UTC/fallback behavior in FR-09 rather than aborting the lookup. (Source: `/home/parallels/snowraven/frontend/src/lib/tauri/weatherService.ts:20-83`; `/home/parallels/snowraven/backend/services/ebird.py:5-89`; `/home/parallels/snowraven/frontend/src/lib/tide.ts:219-222`.)
- **FR-06** — The popup shall resolve checklist coordinates using SnowRaven's exact fallback chain, stopping at the first source that yields both a latitude and longitude: (1) PRIMARY — `ref/region/info/{locId}` bounding-box CENTRE: `lat = (minY + maxY) / 2`, `lng = (minX + maxX) / 2`, only when the response is OK with a non-empty body containing `bounds` with all of `minX, maxX, minY, maxY`; (2) FALLBACK — `product/lists/{locId}?maxResults=1`, reading `lat`/`latitude` and `lng`/`longitude`/`lon` from the first element's `loc`/`location`; (3) LAST RESORT — `data/obs/{locId}/recent?back=365`, taking `lat`/`lng` from the first observation. The popup shall NOT use the page's map pin. (Source: `weatherService.ts:33-70`; `regionInfo.ts:20-35`; `ebird.py:27-79`.)
- **FR-06a** — When the checklist view returns 200 and the ID resolves but NO source in the FR-06 chain yields both a latitude and longitude, `fetchChecklist` shall throw the resolution-aware error `Could not find coordinates for location {locId}.` (carrying internal `status: 502`). Because both the weather path and the tide path call the same `fetchChecklist`, this single throw shall surface on BOTH blocks: weather shall render the FR-36 mapped message `Could not find coordinates for location {locId}.` in a role=alert block, and tide shall render the `error` state per FR-37/FR-37a (`Tide data unavailable right now.`, a quiet role=status line). Nothing shall be auto-copied. (Source: `weatherService.ts:72-73`; `tideService.ts:59` calling `fetchChecklist`.)
- **FR-07** — The region-info (primary) lookup shall be best-effort: a thrown error, non-OK response, or empty/boundless body shall NOT abort resolution but shall fall through to source 2, then source 3. A failed region-info lookup shall not be cached. Where region-info succeeds and the checklist view supplied no name, the location name shall be taken from region-info (`result || name`); the displayed `loc_name` shall ultimately fall back to `locId` when no name resolves.
- **FR-08** — The popup shall derive the IANA timezone name for the resolved coordinates in-browser (the extension has no native timezone-finder) and shall use that single timezone name for BOTH (a) interpreting `obsDt` as wall-clock local time when computing weather/tide start/end timestamps and (b) formatting local times (sunrise/sunset and any local-time rendering). When no timezone can be determined, the popup shall fall back to `'UTC'`, matching SnowRaven's `... or "UTC"` default, so weather still formats rather than failing. (Source: `weatherService.ts:92-127,150`; `weatherFormatter.ts:77-89`; `backend/formatters/weather.py:70-72`.)
- **FR-08a** — The chosen in-browser timezone mechanism shall agree with the desktop's `timezonefinder`/`tzf-rs` output on a fixed acceptance set of checklist coordinates that includes (a) interior points, (b) DST-boundary dates, and (c) coordinates within a few kilometres of an IANA timezone border, such that the resolved zone produces ZERO byte difference versus the golden for both the selected OpenWeather hour and the formatted sunrise/sunset. Because the zone is required BEFORE the OpenWeather fetch (to compute `startTs`/`endTs` and thus select which hour is fetched), reading OpenWeather's response-root `timezone` field is NOT an acceptable mechanism (it arrives only after the fetch) and shall not be used to pick the hour. The acceptance set and zero-byte-difference bound are verified at Stage 5 against the golden. (Source: Open Question 1; `weatherService.ts:94-127,150-156`.)
- **FR-09** — The popup shall compute `startTs` by parsing `obsDt` as local time in the resolved timezone using SnowRaven's DST-aware iterative `parseLocalDateTimeInZone` convergence (`startTs = floor(obsDtMs / 1000)`), and `endTs` by adding `durationHrs` hours (`durationMs = (durationHrs || 1) * 3600 * 1000`; `endTs = floor((obsDtMs + durationMs) / 1000)`). A date-only `obsDt` shall be normalized to local midnight (`... 00:00`) before parsing. `durationHrs` may be fractional. A falsy or zero `durationHrs` shall resolve to `1` (per `(durationHrs || 1)`); a negative `durationHrs` is out of contract and shall be treated as `1` (clamped to the default), so `endTs >= startTs` always holds. (Source: `weatherService.ts:94-127,150-156`; `weather.py:36-45`.)

### Weather Fetch and Formatting (OpenWeather One Call 3.0 `timemachine`)

- **FR-10** — The popup shall issue ONE `timemachine` call when `startTs === endTs`, and TWO calls (at `startTs` then `endTs`) when they differ; calls shall be issued concurrently and the resulting one or two `HourlyResponse` objects passed to the formatter in `[start, end]` order, so `responses[0].data[0]` is the canonical start hour. (Source: `weatherService.ts:157-161`; `weather.py:45-50`.)
- **FR-11** — Each weather request shall GET `https://api.openweathermap.org/data/3.0/onecall/timemachine` with query params `lat`, `lon`, `dt` (Unix seconds), `appid` (the user's OpenWeather key), and `units=imperial`. Only `data[0]` of each response is consumed. (Source: `weatherService.ts:85-90`; `backend/services/openweather.py:5-18`.)
- **FR-12** — Units shall be imperial throughout: temperature and dew point in °F, wind speed in mph (for the Beaufort table), cloud cover and humidity in %. No client-side unit conversion shall be performed — the API returns imperial values directly.
- **FR-13** — The formatted weather block shall be a fixed 10-line body followed by the attribution line — 11 lines total, joined by single `\n` with NO blank lines and NO trailing newline, in this exact order: (1) condition emoji; (2) capitalized condition text; (3) `Temperature: <range>°F`; (4) `Wind: <beaufort words>`; (5) `Wind Direction: <cardinals>`; (6) `Cloud Cover: <range>%`; (7) `Humidity: <range>%`; (8) `Dew point: <range>°F` (note lowercase "point"); (9) `Sunrise: <12h local>`; (10) `Sunset: <12h local>`; (11) the literal attribution line `Weather generated by <a href="https://github.com/dtgibson/snowraven">SnowRaven</a>`. The HTML `<a>` tag is load-bearing and shall not be escaped or altered. The standalone weather block contains no blank line. (Source: `weatherFormatter.ts:33,128-144`; `weather.py:24,105-117`.)
- **FR-14** — Numeric ranges (temperature, cloud cover, humidity, dew point) shall round each contributing value with banker's rounding (round-half-to-even via `bankersRound`, NOT `Math.round`), take min/max of the rounded set, and render `<lo><unit>` when `lo == hi` else `<lo> - <hi><unit>` (separator is space-hyphen-space). Lo and hi shall be derived AFTER rounding. (Source: `weatherFormatter.ts:37-42,69-75`.)
- **FR-15** — The Wind line shall be the deduplicated set of Beaufort words across all responses, ordered lightest-to-strongest by the fixed Beaufort scale (≤1 Calm, ≤3 Mostly calm, ≤7 Light breeze, ≤12 Gentle breeze, ≤18 Moderate breeze, ≤24 Fresh breeze, ≤31 Strong breeze, ≤38 Near gale, else Gale; comparison is inclusive `<=`), joined by ` - `. (Source: `weatherFormatter.ts:15-25,44-49,108-110`.)
- **FR-16** — The Wind Direction line shall be the cardinal directions across all responses in first-seen (insertion) order with duplicates removed (NOT sorted), joined by ` - `, where `cardinal(deg) = CARDINALS[bankersRound(deg / 45) % 8]` over `['N','NE','E','SE','S','SW','W','NW']`. (Source: `weatherFormatter.ts:31,51-53,112-120`.)
- **FR-17** — The condition emoji (line 1) and condition text (line 2) shall be derived ONLY from `responses[0].data[0]` (the start hour), not merged across responses. The emoji maps OpenWeather condition-id ranges (200–232 ⛈️, 300–321 🌦️, 500–531 🌧️, 600–622 ❄️, 700–781 🌫️, 800 ☀️, 801 🌤️, 802 ⛅, 803 🌥️, 804 ☁️, else 🌡️); the condition text is `description` lowercased then first-character uppercased. (Source: `weatherFormatter.ts:55-67,91-95,123-124`.)
- **FR-18** — Sunrise and Sunset (lines 9–10) shall be taken from `responses[0].data[0].sunrise`/`.sunset` (Unix seconds) and rendered as 12-hour local time in the resolved timezone via `Intl.DateTimeFormat`, formatted `h:mmam`/`h:mmpm` (no leading zero on hour, 2-digit minute, lowercase am/pm, no space). The second response's sunrise/sunset shall be ignored. (Source: `weatherFormatter.ts:77-89,125-126`.)
- **FR-19** — Weather worked example (single hour, `America/New_York`): OpenWeather data `temp=54.3, humidity=89, dew_point=51.5, wind_speed=8.3, wind_deg=270, clouds=100, weather id=804 'overcast clouds', sunrise=1714554480, sunset=1714603980` shall format to exactly (real newlines, no blank lines, no trailing newline): `☁️\nOvercast clouds\nTemperature: 54°F\nWind: Gentle breeze\nWind Direction: W\nCloud Cover: 100%\nHumidity: 89%\nDew point: 52°F\nSunrise: 5:08am\nSunset: 6:53pm\nWeather generated by <a href="https://github.com/dtgibson/snowraven">SnowRaven</a>`. (Source: `weatherFormatter.golden.py:118-121,191`.)

### Tide Fetch and Formatting (NOAA CO-OPS, keyless)

- **FR-20** — The tide path shall reuse SnowRaven's tide TypeScript modules byte-for-byte: `tide.ts`, `tideStations.ts`, `tideNotice.ts`, `tideFormatter.ts` (which imports `ATTRIBUTION` from `weatherFormatter.ts`), and the bundled station asset. No tide logic shall be reimplemented or reinterpreted. The runtime path adapts the desktop tide-service logic to call NOAA directly from the popup via standard `fetch`. The Python backend twins are reference-only and shall not ship. (Source: `tide.ts:1-242`; `tideStations.ts:1-93`; `tideNotice.ts:1-19`; `tideFormatter.ts:1-65`; `tauri/tideService.ts:50-95`.)
- **FR-21** — The tide path shall reuse the same eBird resolution (FR-05–FR-09) for coordinates, `obs_dt`, and `duration_hrs` (default 1) — tide and weather share one lookup and the API-resolved coordinates, never the page map pin. Because the tide service calls the same `fetchChecklist`, a checklist-resolution throw (404, non-OK eBird, or no coordinates) escapes the tide service before the `getJson` seam and surfaces as the tide `error` state per FR-37a/FR-06a.
- **FR-22** — The popup shall find the nearest NOAA station by haversine (R = 3958.8 mi) over the bundled US-only station list, taking the genuinely nearest station with NO observation bias (`nearestStation` called with no options). It shall return `null` only when the bundle is empty. (Source: `tideStations.ts:41-78`; `tideService.ts:61`.)
- **FR-23** — The popup shall classify the location via `classifyTideLocation`: `outside-us` first (when the point is in none of the 7 coarse US bounding boxes per `isInUS`), then `too-far` if there is no nearest station or its distance exceeds `TIDE_MAX_MILES` (25 mi), else `ok`. (Source: `tideStations.ts:22,27-39,84-93`.)
- **FR-24** — When status is `ok` (or when an override forces the fetch), the popup shall issue exactly three concurrent GET requests against `https://api.tidesandcurrents.noaa.gov/api/prod/datagetter`, all sharing `datum=MLLW`, `units=english`, `time_zone=lst_ldt`, `format=json`, `application=SnowRaven` (keyless): (1) `water_level` over `[start, end]`; (2) `predictions interval=6` over `[start, end]`; (3) `predictions interval=hilo` over the ±24h-expanded window `[start−24h, end+24h]`. `start = normalizeObsDt(obs_dt)`; `end = shiftLocal(start, duration_hrs || 1)`; dates use `toNoaaDate` (`YYYYMMDD HH:MM`). A falsy/zero `duration_hrs` resolves to `1`; a negative duration is out of contract and clamped to `1` (per FR-09), so `end >= start`. (Source: `tideService.ts:17,30-39,69-78`; `noaa.py:9-32`; `tide.ts:198-241`.)
- **FR-25** — The popup shall treat a NOAA response as an error when its JSON body has a top-level `error` key, regardless of HTTP status (NOAA returns HTTP 200 with an error body for no-data/throttle). The three responses shall be parsed with `parseObserved`/`parsePredictions`/`parseHiLo`, which return `[]` for an error or malformed body and skip non-finite values; `getJson` shall return `null` on any fetch/parse throw. A NOAA error body or `getJson` null therefore degrades to `unavailable` (via empty parsed data → `computeTideReading` returning `null`), NOT to the tide `error` state. (Source: `tide.ts:41-82`; `tideService.ts:41-48`.)
- **FR-26** — The popup shall compute a single `TideReading` via `computeTideReading` using the exact level-source priority: (1) observed gauge points in-window → source `Observed`; (2) continuous predictions in-window → source `Predicted`; (3) interpolated from the hilo curve at start/end plus any in-window extreme → `Predicted`; (4) nearest single point when no in-window samples and no curve to interpolate. `levelMin`/`levelMax` are min/max over the chosen sample set (single value when equal). It shall return `null` only when no usable level data exists (→ status `unavailable`). (Source: `tide.ts:121-195`; `tide.py:117-173`.)
- **FR-27** — The popup shall derive trend as: rising if `endV − startV > 0`, falling if `< 0`; if flat (`== 0`), infer from brackets (toward next High = rising, next Low = falling; else previous High ⇒ falling, previous Low ⇒ rising; else default rising). It shall set `turnedDuring = true` when any high/low event falls within `[start, end]`. The trend renders via `trendLine` as the literal byte string `Tide: Rising` or `Tide: Falling` (the `Tide: ` prefix and the capitalized first letter of the trend word are load-bearing), and `Tide: Rising (turned during your checklist)` / `Tide: Falling (turned during your checklist)` when `turnedDuring` is true. Previous/next high-low are labeled with `kind` and a 12-hour `timeLocal` via `clockTime`. (Source: `tide.ts:173-194`; `tideFormatter.ts:23-26`.)
- **FR-28** — The standalone tide block (`formatTide`) shall be emitted byte-for-byte, joined by single `\n` with NO blank lines, in this exact order: `🌊`; `Observed` or `Predicted`; `Water level: <min> ft` or `Water level: <min> – <max> ft`; the trend line `Tide: <Rising|Falling>` (with the ` (turned during your checklist)` suffix when `turnedDuring`, per FR-27); (if present) `Previous <high|low>: <v> ft at <time>`; (if present) `Next <high|low>: <v> ft at <time>`; `Station: <name> (<id>), <dist> mi away`; `Relative to MLLW`; then `Tide data from NOAA CO-OPS · via <a href="https://github.com/dtgibson/snowraven">SnowRaven</a>`. The level range separator is exactly ` – ` (space, U+2013 EN DASH, space). Levels round to 1 decimal via `ft()`; distance uses `toFixed(1)`. The NOAA credit stays plain text; the SnowRaven credit is the HTML anchor. (Source: `tideFormatter.ts:13-54`; `tide.ts:210-212`.)
- **FR-28a** — Tide worked example (mirrors FR-19; `formatTide` byte string). Given a `TideReading` with `source = 'observed'`, `levelMin = 2.34`, `levelMax = 5.61`, `trend = 'rising'`, `turnedDuring = true`, `prevHL = { kind: 'low', v: 1.82, timeLocal: '7:42am' }`, `nextHL = { kind: 'high', v: 5.61, timeLocal: '1:58pm' }`, and `station = { name: 'The Battery', id: '8518750' }` at `distanceMi = 3.4`, `formatTide` shall emit exactly (real newlines, no blank lines, no trailing newline): `🌊\nObserved\nWater level: 2.3 – 5.6 ft\nTide: Rising (turned during your checklist)\nPrevious low: 1.8 ft at 7:42am\nNext high: 5.6 ft at 1:58pm\nStation: The Battery (8518750), 3.4 mi away\nRelative to MLLW\nTide data from NOAA CO-OPS · via <a href="https://github.com/dtgibson/snowraven">SnowRaven</a>`. The U+2013 EN DASH in the level range and the literal `Tide: Rising (turned during your checklist)` trend line are load-bearing. (Source: `tideFormatter.ts:18-54`; `tide.ts:208-212`.)
- **FR-29** — The bundled NOAA station list shall be the single canonical asset produced by SnowRaven's `build-tide-stations.mjs` (`noaa-tide-stations.json`, ~3,241 US stations, ~346 KB), trimmed to exactly the six fields the lookup uses (`id, name, lat, lng, state, obs`), with US states + territories only (Great Lakes and foreign stations excluded). It is the only large asset; the popup shall consume it unchanged and shall NOT fetch the station list at runtime. The coarse US boxes used by `isInUS` must match the build script's boxes. (Source: `build-tide-stations.mjs:1-107`; `tideStations.ts:7,9-17,19,27-39`.)

### Copy Behavior

- **FR-30** — On weather success the popup shall auto-copy ONLY the formatted weather block (the `formatWeather` output, ending in the attribution line) to the system clipboard via `navigator.clipboard.writeText`, with the legacy `textarea`+`execCommand` fallback from `copyText`. The popup shall hold the `clipboardWrite` permission so the copy succeeds reliably even though it follows the awaited lookup (the permission removes the transient-activation requirement). Auto-copy shall fire once per popup open and only on weather success. Tide shall NEVER be auto-copied. (Source: `clipboard.ts:16-49`; `weatherFormatter.ts:33,142-144`; strategic-brief.md:14,26,32.)
- **FR-31** — The popup shall structure the flow so the weather clipboard write occurs within the gesture's user-activation window (the clipboard write follows async network work, and SnowRaven's clipboard seam documents that a post-`await` web clipboard write loses user activation and throws `NotAllowedError`; `clipboard.ts:3-11`). If the clipboard write fails (activation lost or `execCommand` returns false), `copyText` shall return `false` rather than throw, the auto-copy shall fail silently without flipping any button to "Copied!", and the weather block shall remain visible and selectable so the user can use its manual Copy button. On auto-copy failure the polite live region shall NOT announce "copied"; the accessibility outcome is governed by FR-48a.
- **FR-32** — The popup shall give the weather block and the tide block each its OWN Copy button. The weather Copy button copies the standalone `formatWeather` string; the tide Copy button copies the standalone `formatTide` string (the variant ending in `· via SnowRaven`). The tide Copy button shall be the ONLY way tide reaches the clipboard, and it shall be active only in the tide `ok` state. The popup shall NOT render a combined "weather & tide together" button. (Source: `WeatherTidePanel.tsx:175,200-203`; `WeatherTideSection.tsx:91-93`.)
- **FR-33** — Each Copy button shall, on activation, write its block's text to the clipboard, flip its label/icon to "Copied!" for approximately 2000 ms, then revert. Each block's copied state shall be independent. (Source: `WeatherTidePanel.tsx:51-80,57`.)

### Popup UI and States

- **FR-34** — The popup shall render the weather block and the tide block (or its state notice) in a monospace `<pre>` with `white-space: pre` and horizontal overflow scrolling, so wrapping never alters the bytes, preserving every newline. The displayed weather text equals `formatWeather` output; the displayed tide text equals `formatTide` output (each a single-`\n`-joined block with no blank lines). The HTML `<a>` attribution appears as literal angle-bracket text on screen and is what gets copied. All colors shall come from `var(--sr-*)` tokens. (Source: `WeatherTidePanel.tsx:34-48`.)
- **FR-35** — Weather and tide shall load independently: one block failing shall not prevent the other from rendering. The popup shall show a single combined loading row (role=status, "Loading weather & tide…") only while nothing has resolved yet; once either resolves, it shall render the resolved block and a per-block loading row (role=status, "Loading weather…" / "Loading tide…") for whichever is still pending. Spinners shall be aria-hidden with the textual status carrying the announcement. (Source: `WeatherTidePanel.tsx:147-181,204-208`; `WeatherTideSection.tsx:32-62`.)
- **FR-36** — The popup shall render a weather error as a role=alert block carrying the resolution-aware message. The mapping keys off the eBird RESPONSE status (matching `weatherService.ts:23-25`), not the thrown app-error's remapped numeric status, mapping the known failures to their exact SnowRaven strings: eBird `res.status === 404` → `Checklist not found. Check the ID and try again.` (thrown app-error carries internal `status: 400`); eBird `res.status === 401` or `403` (invalid/expired/forbidden key) → `eBird API key invalid. Check it in Settings.`; any other non-OK eBird (`res.ok === false`, including `429`) → `Could not fetch checklist data. Please try again.`; OpenWeather `401`/`403` (invalid/expired key) → `OpenWeather API key invalid. Check it in Settings.`; any other non-OK OpenWeather (including `429`) → `Weather data unavailable for this checklist.`; no coordinates resolvable → `Could not find coordinates for location {locId}.`; missing key (not yet configured) → `API key not configured. Add it in Settings.`; otherwise → `Something went wrong. Please try again.`. The thrown app-error carries an internal status (400/500/502) for transport bookkeeping, so downstream code shall key its user-facing message off the error message/detail, not the numeric thrown status. The `401`/`403` key-invalid messages point the user to Settings; `429` (rate limit) deliberately falls behind the generic "try again" messages as a transient case. On weather error, nothing shall be auto-copied. (Source: `weatherService.ts:23-25,73,88,142-147`; `WeatherTideSection.tsx:32-41`.)

- **FR-37** — The popup shall render the tide states. The ported tide SERVICE returns exactly four statuses — `ok`, `too-far`, `outside-us`, `unavailable` — and a fifth UI-only status `error` is set when the tide service call THROWS (see FR-37a). Rendering: `ok` → the tide block plus its Copy button; `too-far`/`outside-us` → an amber notice plus a one-tap override button; `unavailable` → a quiet role=status line `No tide reading available.`; `error` → a quiet role=status line `Tide data unavailable right now.`. Tide `unavailable` and `error` shall never render as alerts and shall never block the weather block. (Source: `WeatherTidePanel.tsx:209-236`; `tide.ts:31-37`; `WeatherTideSection.tsx:60`.)
- **FR-37a** — The tide `error` state is a documented divergence from the ported service's status union (which is `'ok' | 'too-far' | 'outside-us' | 'unavailable'`), added at the UI layer (mirroring desktop `WeatherTideSection.tsx:60`'s `catch { set({ status: 'error' }) }`). It is produced ONLY when the tide service call itself throws — i.e. an exception escapes `getTide` BEFORE the `getJson` seam (where NOAA throws are already swallowed to `null`/`unavailable` per FR-25). The reachable triggers are: (1) a missing eBird key (`getTide` throws before any fetch), and (2) any `fetchChecklist` throw — eBird `404` (`Checklist not found.`), non-OK eBird, or no-coordinates (`Could not find coordinates for location {locId}.`, per FR-06a). NOAA-side failures (error body, fetch/parse throw, NOAA host-permission denial) do NOT produce `error`; they degrade to `unavailable` per FR-25/FR-44. The UI `TideStatus` type shall be declared as `'ok' | 'too-far' | 'outside-us' | 'unavailable' | 'error'` to make this divergence explicit. (Source: `WeatherTideSection.tsx:55-61`; `tideService.ts:50-95`; `weatherService.ts:23-25,72-73`.)
- **FR-38** — For `too-far`/`outside-us`, the popup shall show the amber notice from `tideTooFarNotice(station, distanceMi, kind)` with an override button labeled by `tideOverrideLabel(kind)`, byte-identical to the shared strings: too-far → `The nearest tide station is <mi> miles away (<station>). Tide data may not reflect your spot.` with button `Show it anyway`; outside-us → `Tide information is only available in the US. The nearest US station is <station>, <mi> miles away.` with button `Show nearest US station`. The notice distance uses `toFixed(0)` (the block uses `toFixed(1)`). (Source: `tideNotice.ts:9-19`.)
- **FR-39** — Clicking the override button shall re-run the tide lookup with `force = true`, which skips the classify gate and fetches the same nearest station's data anyway (no re-selection). An overridden `ok` result shall be shown with the per-block tide Copy button (still never auto-copied). (Source: `tideService.ts:64-67`; `tide.ts:214-218`.)

### Keys, Options Page, and First-Run Gating

- **FR-40** — The extension shall provide an Options page with exactly two locally-stored, masked/secret-style text fields — `OpenWeather API key` and `eBird API key` — each accompanied by a visible labeled link to obtain the free key (OpenWeather: openweathermap.org API keys; eBird: ebird.org/api/keygen), a save affordance, and a clear save confirmation. Keys shall be persisted via `chrome.storage.local` and shall never leave the device except as auth on the direct eBird (`X-eBirdApiToken` header) and OpenWeather (`appid` query param) calls. No bundled, shared, or default key shall ship. (Source: `weatherService.ts:21,86,137-147`; product-brief.md:29.)
- **FR-41** — On every popup open the extension shall resolve key status before acting. It shall NOT attempt a lookup or show a false "keys missing" state while status is still resolving (a still-resolving / `null` status is treated as resolving, never missing). When a required key is missing, it shall show one nudge per missing key, naming it (eBird and/or OpenWeather), with a "Go to Settings →" action that opens the Options page via `chrome.runtime.openOptionsPage`, and shall not fetch, request host permissions, or auto-copy until the required keys are present. (Source: `WeatherTideSection.tsx:106-111,138-148`.)
- **FR-42** — Key gating shall reflect what each service needs: a missing eBird key shall block BOTH weather and tide (eBird resolves the checklist for both); a missing OpenWeather key shall block weather but tide MAY still proceed (NOAA is keyless). A missing key (not-yet-configured) is the first-run nudge of FR-41; an invalid/expired key (a key IS configured but the API returns `401`/`403`) is distinct and is surfaced by FR-36's key-invalid weather message and, on the tide side, by the `error` state when the throw originates in eBird resolution. (Source: `weatherService.ts:136-147`; `backend/routers/tide.py:23-24`.) **v1 scope (accepted at QA):** when either key is missing the popup shows the single first-run missing-keys nudge naming the absent key(s); showing tide on its own when only OpenWeather is missing is a deferred enhancement.

### Permissions, Privacy, and Cross-Browser

- **FR-43** — Host access to `api.ebird.org`, `api.openweathermap.org`, and `api.tidesandcurrents.noaa.gov` (and no other hosts) shall be declared only under `optional_host_permissions` and requested at runtime via `chrome.permissions.request` from a user gesture; the extension shall hold no host permission until the user grants it. Per FR-43a the three origins shall be requested TOGETHER in a single `chrome.permissions.request` on the first lookup click. (Source: `weatherService.ts:7-8`; `tideService.ts:17`.)
- **FR-43a** — Host-permission request granularity is locked (no longer an Open Question): on the first lookup click, the popup shall issue ONE combined `chrome.permissions.request` listing all three origins together, so a single grant covers weather and tide. The extension shall NOT request the origins individually or per-API. Consequently the only two outcomes are: (a) all three origins granted → the lookup proceeds; (b) the combined request denied → the whole lookup is blocked and FR-44 applies. There is no per-origin partial-grant state. (Source: Open Question "Host-permission request granularity"; `weatherService.ts:7-8`.)
- **FR-44** — When the combined host-permission request (FR-43a) is denied or has not yet been granted, the popup shall surface a clear, recoverable in-popup state explaining that API access is required, and let the user retry the grant from a fresh user gesture, rather than failing silently. Because the grant is all-or-nothing (FR-43a), a denial blocks the entire lookup (both weather and tide); neither block fetches. (Note: the degraded "only NOAA denied / only OpenWeather denied" matrix is NOT reachable under the locked combined-request model and is intentionally removed. The NOAA-denied → tide `unavailable` and OpenWeather-denied behaviors described informally elsewhere apply only to a future per-origin model and are out of scope for v1.) (Source: FR-43a; `tideService.ts:17`.)
- **FR-45** — The extension shall make no network request that is not one of the three documented direct API calls; it shall contain no backend, proxy, telemetry, analytics, crash reporting, remote config, or usage tracking. No checklist ID, coordinate, timestamp, or comment text shall be sent anywhere except as parameters of those three direct API requests. (Source: product-brief.md:13,27; strategic-brief.md:19.)
- **FR-46** — A single manifest and codebase shall ship to both Chrome and Firefox and behave identically (same popup, auto-copy-weather behavior, tide states, byte-for-byte output, and on-demand combined host-permission request), with no per-browser source forks. The manifest shall include `browser_specific_settings.gecko.id` and `browser_specific_settings.gecko.strict_min_version` for Firefox MV3; Chrome shall ignore these settings. The `strict_min_version` shall be pinned per FR-46a to a Firefox version that is confirmed to support BOTH runtime `optional_host_permissions` requests via `chrome.permissions.request` AND `navigator.clipboard.writeText` within a popup user-activation window. (Source: strategic-brief.md:16,28,43.)
- **FR-46a** — Determining the Firefox `strict_min_version` floor and `gecko.id` form is a BLOCKING Stage-3 verification (promoted out of Open Questions). The pinned `strict_min_version` shall be the lowest Firefox version empirically confirmed to support, under MV3, BOTH (a) a runtime `chrome.permissions.request` of `optional_host_permissions` from a user gesture and (b) a `navigator.clipboard.writeText` succeeding within the popup's user-activation window. If a candidate floor fails either, it shall be raised until both pass. The full grant-then-auto-copy flow shall be exercised on the pinned Firefox version (not merely "loads in Firefox") and is verified by QA-53a. (Source: Open Question "Firefox gecko.id / strict_min_version"; FR-46; NFR-07.)

## Non-Functional Requirements

- **NFR-01 — Permissions:** The extension shall ship as a popup-only Manifest V3 extension with an `action` `default_popup` as its sole runtime surface, no content script, and no background/service worker; the `permissions` array shall contain exactly `["activeTab", "clipboardWrite", "storage"]` (`clipboardWrite` so the post-lookup weather auto-copy is reliable — FR-30; `storage` because the two BYO keys live in `chrome.storage.local`, which is `undefined` without it — FR-40) and no host permission, with all host access confined to `optional_host_permissions` per FR-43. (Mirrors FR-01, FR-30, FR-40, FR-43, FR-46; strategic-brief.md:28,42. Verified by QA-54.)
- **NFR-02 — Privacy:** The extension shall run entirely on the user's device with no backend, proxy, hosted service, telemetry, or analytics; the two stored API keys shall reside only in `chrome.storage.local` and shall be transmitted only to their respective API host. (Mirrors FR-40, FR-45; strategic-brief.md:18,19,40. Verified by QA-40 and QA-45.)
- **NFR-03 — Accessibility:** The popup and Options page shall meet WCAG 2.1 AA — full keyboard operability, descriptive accessible names, polite-live announcement of results/errors/"Copied!"/auto-copy success-and-failure, visible focus, AA contrast in both themes, OS-driven dark theme, and reduced-motion respect. (Mirrors FR-47–FR-49, FR-48a; product-brief.md:24. Verified by QA-47, QA-48, QA-48a, QA-49.)
- **NFR-04 — Bundle size:** The shipped bundle shall be kept as small as possible, carrying only the dependencies in FR-50 (no component library; `maplibre-gl`/`react-map-gl`/`recharts`/`@tauri-apps/*`/`commentBlocks.ts` excluded from any runtime path); the trimmed NOAA station JSON (~346 KB, six fields) shall be the only large asset, and the in-browser timezone mechanism shall be minimized and counted against the budget. (Mirrors FR-29, FR-50; strategic-brief.md:43. Verified by QA-55.)
- **NFR-05 — Performance:** On a normal network, the popup shall display the formatted weather (and resolve tide where applicable) within approximately 2 seconds of the click; it shall parallelize the OpenWeather and NOAA fetches (as the source does), resolve weather and tide concurrently, walk the eBird fallback chain only when the region-centre primary fails, and never let tide block the weather render. (Source call chains: `weatherService.ts:20-83,157-161`; `tideService.ts:74-78`. Verified by QA-52.)
- **NFR-06 — Performance (clipboard timing):** The auto-copy of the weather block shall land within the user-activation of the triggering gesture; the clipboard write shall not fail for loss of user activation after asynchronous work, and any copy failure shall surface a recoverable manual-copy path rather than a silent loss. (Mirrors FR-30, FR-31, FR-48a; `clipboard.ts:3-11`. Verified by QA-30, QA-31, QA-48a.)
- **NFR-07 — Compatibility:** The extension shall behave identically on the pinned-minimum-and-newer Chrome and Firefox under Manifest V3 — same on-demand combined host-permission flow, same clipboard write within user activation, and byte-for-byte identical formatted output (including the HTML `<a>` attribution) — which requires `Intl.DateTimeFormat` and banker's rounding to produce identical results in both JS engines, and requires the Firefox floor pinned per FR-46a. (Mirrors FR-46, FR-46a; strategic-brief.md:16,41. Verified by QA-53 and QA-53a.)

## Out of Scope

- Auto-copying tide — only weather auto-copies on click; tide is copied solely via its own button (strategic-brief.md:32).
- A combined one-button "weather + tide together" copy — individual buttons cover it; `buildCombined`/`formatTideBody`/`COMBINED_ATTRIBUTION` are retained in the port but unwired in v1, and `buildCombined`'s `\n\n` blank lines never appear in a v1-rendered block (strategic-brief.md:35; FR-51).
- Detecting and skipping a weather/tide block already present in the comment (SnowRaven's duplicate guard via `commentBlocks.ts`) and any reconciliation note about OpenWeather revising historical data — deferred to a later refinement; `commentBlocks.ts` is therefore not a required runtime dependency (FR-50) and the historical non-determinism itself is expected and shall not be treated as an error (strategic-brief.md:33).
- Any lookup outside the eBird checklist edit context — no arbitrary location/time queries and none of the desktop app's other tabs (strategic-brief.md:34; product-brief.md:31).
- Editing or posting to eBird on the user's behalf — the extension reads what it needs and copies to the clipboard; pasting remains the user's action (product-brief.md:30).
- Any backend, proxy, or hosted service, and any bundled or shared API key (product-brief.md:27,29).
- Per-origin host-permission grants and the associated partial-denial degraded matrix — v1 uses a single combined three-origin request (FR-43a), so "only NOAA denied" / "only OpenWeather denied" states are not built in v1 (FR-44).
- Runtime fetching/refresh of the NOAA station list — staleness between releases is accepted; a decommissioned bundled station degrades to `unavailable`, and a newly added real station is matched only after the asset is rebuilt at release (FR-29).

## Open Questions

- **In-browser IANA timezone resolution mechanism (highest-impact HOW).** The WHAT and the acceptance bound are now locked (FR-08, FR-08a): the same in-browser zone feeds BOTH `parseLocalDateTimeInZone` (selecting which OpenWeather hour is fetched) AND `formatLocalTime` (sunrise/sunset), so a divergent zone near a tz border mis-formats local times and can shift the historical hour. The remaining OPEN item is purely the Stage-3 implementation choice. **Default:** ship a trimmed coordinate→IANA lookup (bundled tz-boundary data or a small library) whose output agrees with `timezonefinder` to a zero-byte-difference bound across the FR-08a acceptance set (DST + tz-border cases), falling back to `'UTC'` when no zone is found. Reading OpenWeather's response-root `timezone` is excluded because the zone is needed before the fetch (FR-08a).
- **Exact at-rest key storage.** The desktop used an OS keychain; the extension cannot replicate that. **Default:** `chrome.storage.local` (default, unencrypted) is acceptable for the two bring-your-own keys, consistent with the no-backend model.
- **Numeric NFR thresholds.** Hard ceilings for bundle KB and whether ~2 s is p50 or a hard p95 are not pinned in the briefs. **Default:** treat ~2 s as a p50 target on a normal network and "smallest possible bundle" as a directional constraint (NOAA JSON + timezone data dominate), with concrete ceilings confirmed at Stage 3.
