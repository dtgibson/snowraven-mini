# PRD — Onboarding & Combined Copy
**Feature:** onboarding-combined-copy
**Date:** 2026-06-15
**Stage:** 2 — The Planner
**Source:** strategic-brief.md (approved)

## Feature Overview
Two popup UX additions shipped in one lane: a first-run walkthrough that
replaces today's bare missing-keys notice with a guided path from install to
first lookup, and a "Copy weather & tide together" button that copies a single
block byte-for-byte identical to SnowRaven desktop's `buildCombined()` output.
Both are additive — no existing weather, tide, copy, or auto-copy behavior
changes.

## User Stories

> **US-01** — As a first-time birder who just installed the extension, I want
> the popup to tell me which two keys I need and where to get them, so that I
> can reach my first weather copy without leaving the popup confused.

> **US-02** — As a new user reading the walkthrough, I want to open Settings
> directly from it, so that I can paste my keys without hunting for the options
> page.

> **US-03** — As a screen-reader or keyboard user setting up for the first
> time, I want the walkthrough to be fully reachable and announced, so that the
> guided path works the same for me as for a sighted mouse user.

> **US-04** — As a birder editing a checklist that has both weather and tide, I
> want one button that copies them together, so that I don't have to copy two
> blocks and stitch them by hand with the right spacing and attribution.

> **US-05** — As a birder who trusts SnowRaven's output, I want the combined
> block to match the desktop app exactly, so that my checklist comments read
> the same whether I used the app or the extension.

> **US-06** — As a birder who only wants the weather, I want the combined
> button to stay out of my way when there's no tide, and I want nothing about
> the existing auto-copy or per-block buttons to change, so that my current
> one-click habit is untouched.

## Functional Requirements

### First-run walkthrough

> **FR-01** — The popup shall show the first-run walkthrough in place of the
> current bare missing-keys notice whenever it opens on an eBird checklist page
> with one or both required keys (eBird, OpenWeather) unset.

> **FR-02** — The walkthrough shall name both required keys explicitly: the
> eBird API key and the OpenWeather API key.

> **FR-03** — The walkthrough shall state that both keys are free to obtain.

> **FR-04** — The walkthrough shall point the user to where each key is
> obtained. Where it links out, the link shall open in a new tab and shall not
> request any new permission.

> **FR-05** — The walkthrough shall provide a control that opens the Settings
> (Options) page, reusing the existing `openOptions()` path.

> **FR-06** — When exactly one key is missing, the walkthrough shall make clear
> which key is still needed and shall indicate that the other key is already
> set, so a returning half-configured user is not told to re-enter a key they
> already have.

> **FR-07** — The walkthrough shall not request, store, or transmit any key
> itself; key entry remains solely on the Options page via the existing
> `KeyField` inputs and `chrome.storage.local`.

> **FR-08** — After the user adds the missing key(s) on Settings and reopens
> the popup on a checklist, the popup shall proceed to the normal lookup flow
> and shall not show the walkthrough.

> **FR-09** — The walkthrough shall replace only the `missing-keys` page state.
> The `not-on-checklist`, `checklist-view`, `permission-blocked`, `loading`,
> and `result` states shall be unchanged.

### Combined copy button

> **FR-10** — The popup shall render a "Copy weather & tide together" button
> only when both the weather block and the tide block are present and
> successful in the same open (weather `status: 'success'` and tide
> `status: 'ok'`).

> **FR-11** — When weather is missing/errored, or tide is in any state other
> than `ok` (`loading`, `too-far`, `outside-us`, `unavailable`, `error`), the
> combined button shall not render.

> **FR-12** — The combined button shall be additive: the existing per-block
> "Copy" buttons on the Weather and Tide slots shall remain present and
> unchanged.

> **FR-13** — The button's visible label shall be exactly
> `Copy weather & tide together`.

> **FR-14** — Clicking the combined button shall place on the clipboard a
> string equal to `buildCombined(weatherFormatted, tideBody)`, where
> `weatherFormatted` is the successful weather block's `formatted` string and
> `tideBody` is the tide reading's `body` string (`formatTideBody` output, which
> keeps the inline NOAA credit and ends with the bare NOAA credit line).

> **FR-15** — The combined block shall equal SnowRaven desktop's
> `buildCombined()` output byte-for-byte: the weather block with its standalone
> `Weather generated by …SnowRaven…` attribution line removed, then a blank
> line, then the tide body, then a blank line, then the single combined
> attribution `Weather and tide generated by <a href="https://github.com/dtgibson/snowraven">SnowRaven</a>`.
> This parity shall be locked by a golden test asserting the exact string with
> `.toBe`.

> **FR-16** — The combined copy shall be manual only. The popup shall not
> auto-copy the combined block on open or on any state change; it copies solely
> in response to the user activating the combined button.

> **FR-17** — The existing weather-only auto-copy on popup open shall be
> unchanged in trigger, content, timing, and its "Copied to clipboard"
> confirmation.

> **FR-18** — On a successful combined copy, the button shall give the same
> transient visual confirmation pattern as the existing per-block Copy buttons
> (a "Copied!" affordance that reverts after roughly 2000 ms).

> **FR-19** — If the clipboard write fails (`copyText` returns false), the
> button shall not show the success confirmation, and shall not claim the copy
> succeeded.

> **FR-20** — Night weather blocks shall flow through the combined copy
> unchanged, preserving the unspaced moon-phase emoji run (e.g. `☁️🌗`) exactly
> as in the standalone weather block.

> **FR-21** — The combined block shall carry exactly one SnowRaven attribution
> (the combined attribution at the end) and no standalone weather attribution
> line.

### Threading tide body to the UI

> **FR-22** — The tide UI state shall carry the tide `body` string alongside
> `formatted` for the `ok` case, so the combined copy can be assembled from the
> body (which keeps the inline NOAA credit) rather than from the standalone tide
> block (which appends `· via SnowRaven`). The tide service already returns
> `body`; this requirement covers exposing it through the UI state to the panel.

## Non-Functional Requirements

> **NFR-01 — Accessibility:** The walkthrough and the combined button shall
> meet WCAG 2.1 AA: keyboard-reachable with visible focus, properly labelled,
> and AA contrast in both light and dark themes.

> **NFR-02 — Accessibility (announcements):** Async state changes shall be
> announced through the existing shared polite live region (`announce(...)`):
> the walkthrough's state on open, and the combined copy's success
> confirmation. Announcements shall not duplicate or clobber the existing
> weather/tide announcements.

> **NFR-03 — Parity:** Combined output parity is a hard contract verified by a
> golden `.toBe` test (FR-15). `npm test` and `npm run build` shall be green
> before ship.

> **NFR-04 — Permissions:** No new permissions. No new `permissions` or
> `optional_host_permissions` entries; the walkthrough's outbound links open in
> a tab and request nothing.

> **NFR-05 — Footprint:** No backend, no content script, no background worker,
> no new runtime dependency. Bundle growth limited to the UI these two
> additions require.

> **NFR-06 — Cross-browser:** Both additions shall behave identically in Chrome
> and Firefox from the single shared codebase and build.

> **NFR-07 — Reduced motion:** Any transient affordance (the combined button's
> "Copied!" state) shall respect the user's reduced-motion preference,
> consistent with the existing copy buttons.

## Out of Scope
- Auto-copying the combined block. SnowRaven never auto-copies it, and neither
  will Mini; it stays a deliberate user action. The weather-only auto-copy on
  open is unchanged.
- The duplicate-block guard (skipping auto-copy when the comment already
  contains a weather block) — separate roadmap item.
- Tide without an OpenWeather key (showing tide on its own when only OpenWeather
  is missing) — separate roadmap item; the walkthrough still requires both keys
  before lookup.
- Any combined copy or onboarding change on the marketing site — the site is
  untouched.
- New weather or tide logic, new permissions, or any backend.
- Re-implementing or re-porting `buildCombined`. The formatter, its constants,
  and its golden test already exist in Mini's `tideFormatter`; this lane wires
  the existing function to a button and threads the tide body through the UI
  state. (If review wants the existing combined golden test extended to assert
  the full string with `.toBe`, that is in scope under FR-15; re-deriving the
  formatter is not.)

## Open Questions

**OQ-01 — Walkthrough step structure (greenfield, no parity anchor).** How many
discrete steps should the walkthrough present, and does it walk one step at a
time or show the path in a single panel? *Default if unanswered:* a single
guided panel (not a multi-step wizard) that names both free keys, links to where
to get each, and offers "Go to Settings" — the lightest change to today's
`missing-keys` state and the smallest bundle impact. This keeps the popup's
one-glance character.

**OQ-02 — Where the walkthrough links for each key.** Exact destinations for
"where to get the key." *Default if unanswered:* eBird → the eBird API key
request page; OpenWeather → the OpenWeather API keys page. Both open in a new
tab. Final URLs confirmed at build time against each provider's current key
page.

**OQ-03 — Combined button placement and styling.** Where the combined button
sits relative to the two block sections, and whether it is primary or secondary
styling. *Default if unanswered:* a single full-width secondary button below the
Tide slot, visually distinct from the two per-block Copy buttons, shown only
when both blocks succeed. The Architect/Designer settle exact placement; this
PRD fixes only the label (FR-13), the trigger condition (FR-10/11), and the
copied bytes (FR-14/15).

**OQ-04 — Returning user who already has both keys but lands mid-walkthrough.**
Not reachable today (the walkthrough only shows when a key is missing), so no
behavior is specified beyond FR-08. *Default if unanswered:* no special case;
once both keys are set the walkthrough never renders.
